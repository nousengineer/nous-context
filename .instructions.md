# ThinkBrew Development Guidelines

## Code Style & Conventions

### NO EMOJIS

- **NEVER** use emojis in code (backend, frontend, comments, or documentation)
- Replace all emojis with:
  1. SVG components from icon libraries (lucide-react, react-icons, heroicons)
  2. Text descriptions for clarity
  3. Set icon props to `null` as placeholder during development

Example: Instead of "Dashboard", use `<DashboardIcon />` or plain text "Dashboard".

## Architecture

### Monorepo Structure

- pnpm workspace monorepo. Two independent product lines live here:
  1. **ThinkBrew** (AI context sync): `packages/core`, `packages/mcp-server`, `packages/cli`
  2. **Editor LLM bridges** (standalone VS Code extensions): `packages/vscode-llm-server`, `packages/antigravity-llm-server`
- The two editor bridges do NOT depend on `@thinkbrew/core`; they are self-contained.
- TypeORM + SQLite (stored at `~/.thinkbrew/data.sqlite`)
- `synchronize: true` in dev — delete the DB file to reset schema

### Build & Dependencies

- `pnpm build` builds all packages (`pnpm -r build`)
- `pnpm test` runs `scripts/test-integration.js`
- Core must build first (others reference it via `workspace:*`)
- MCP server uses a standalone tsconfig with `module: Node16, moduleResolution: Node16` (required for SDK subpath exports)
- CLI extends `tsconfig.base.json` (commonjs)

### Package Details

- **@thinkbrew/core**: entities, services, database, export (6 formats), validation (zod), crypto, pipeline state machine, file tools
- **@thinkbrew/mcp-server**: MCP stdio server exposing resources, project/context/decision tools, file tools, chat tools. bin: `thinkbrew-mcp`. Workspace sandboxed via `--workspace=<path>` arg or `THINKBREW_WORKSPACE` env
- **@thinkbrew/cli**: Commander.js CLI; bin: `think`. Commands: `project`, `context`, `decision`, `export`, `init`
- **vscode-llm-server**: standalone VS Code extension exposing Copilot models as an Ollama-compatible HTTP API
- **antigravity-llm-server**: standalone VS Code extension exposing Windsurf/Antigravity "Cascade" models as an Ollama-compatible HTTP API (reverse-engineered)

## Known Gotchas

### TypeORM & Reflection

- `emitDecoratorMetadata`: union types like `string | null` reflect as `Object` → must specify explicit column type (e.g., `@Column('text', { nullable: true })`)
- Cascade delete: `onDelete: 'CASCADE'` goes on `@ManyToOne` (child side), not `@OneToMany`

### SQLite Limitations

- SQLite can't ALTER FK constraints — delete DB file and let TypeORM recreate
- Use `synchronize: true` in dev for automatic schema updates

### Package-Specific Notes

- MCP SDK `ResourceTemplate` needed for templated URIs; callback signature is `(uri, variables, extra)` not destructured
- The core `tsconfig.json` excludes several experimental/dead modules (see the `exclude` list) — they are intentionally not part of the stable build
